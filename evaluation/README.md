# CodeGenOpt Evaluation Package

This directory is a self-contained evaluation package for CodeGenOpt. It includes
the evaluation source code, previously generated experimental results, and
aggregate tables used for scientific analysis. The full directory can be shared
as a reviewer artifact.

The included results were produced by earlier experiment runs. They are retained
as audit evidence and can be regenerated by running the same evaluation commands
with the same dataset, model/backend configuration, sample indices, and runtime
environment.

## Current Structure

| Path | Description |
| --- | --- |
| `run_evaluation.py` | Main entry point for running CodeGenOpt on benchmark datasets. |
| `data_loaders.py` | Unified dataset loader for HumanEval, 3LM/EvalPlus Arabic, and local JSONL datasets. |
| `sandbox_tester.py` | Executes generated Python code against benchmark tests in a subprocess sandbox. |
| `metrics.py` | Computes pass@k, syntax validity, latency statistics, and resource summaries. |
| `utils.py` | Creates output directories and records runtime/environment information. |
| `prepare_samples.py` | Generates reproducible fixed-size dataset samples using a fixed random seed. |
| `generate_report.py` | Aggregates existing result summaries into tables, figures, and a Markdown report. |
| `results/` | Previously generated experimental outputs and aggregate tables. |

## Evaluation Workflow

The evaluation pipeline follows this sequence:

1. Load a benchmark dataset using `data_loaders.py`.
2. Optionally restrict the dataset by `--dataset_limit` or `--sample_indices_file`.
3. Run each task through the CodeGenOpt coordinator.
4. Capture the final generated code and intermediate outputs.
5. Check syntax validity.
6. Execute the benchmark tests using `sandbox_tester.py`.
7. Compute pass/fail status, latency, and resource metrics.
8. Save raw traces, summaries, CSV files, and comparison-ready JSON files under
   `evaluation/results/`.

## Supported Dataset Modes

| Dataset argument | Source |
| --- | --- |
| `humaneval` | Loads `openai_humaneval` from Hugging Face unless a local override is provided. |
| `3lm` | Loads `tiiuae/evalplus-arabic` with the `humanevalplus-arabic` configuration unless a local override is provided. |
| other names with `--dataset_local_path` | Loads a local JSONL file directly. |

Each dataset record is expected to provide, where available:

- `task_id`
- `prompt`
- `test`
- `canonical_solution`
- `entry_point`

## Running a New Evaluation

Small smoke test:

```bash
python evaluation/run_evaluation.py \
  --dataset humaneval \
  --dataset_limit 1 \
  --attempts 1 \
  --cloud_llm \
  --disable_optimizer \
  --debug_mode
```

HumanEval cloud run with translator and optimizer enabled:

```bash
python evaluation/run_evaluation.py \
  --dataset humaneval \
  --dataset_limit 50 \
  --attempts 1 \
  --cloud_llm \
  --experiment_name HE-CLOUD-TR_ON-OPT_ON-50
```

HumanEval cloud baseline without translator or optimizer:

```bash
python evaluation/run_evaluation.py \
  --dataset humaneval \
  --dataset_limit 50 \
  --attempts 1 \
  --cloud_llm \
  --disable_translator \
  --disable_optimizer \
  --experiment_name HE-CLOUD-TR_OFF-OPT_OFF-50
```

Arabic 3LM/EvalPlus run:

```bash
python evaluation/run_evaluation.py \
  --dataset 3lm \
  --dataset_limit 50 \
  --attempts 1 \
  --cloud_llm \
  --experiment_name 3LM-CLOUD-TR_ON-OPT_ON-50
```

Local JSONL dataset:

```bash
python evaluation/run_evaluation.py \
  --dataset local \
  --dataset_local_path data/my_local_dataset.jsonl \
  --attempts 1 \
  --experiment_name LOCAL-JSONL-RUN
```

Run with a fixed reproducible sample:

```bash
python evaluation/run_evaluation.py \
  --dataset humaneval \
  --sample_indices_file samples/humaneval_sample_50.json \
  --attempts 1 \
  --cloud_llm \
  --experiment_name HE-CLOUD-SAMPLED-50
```

## Output Files

For an experiment named `EXP_ID`, the runner writes:

| Output | Description |
| --- | --- |
| `results/raw/details_EXP_ID.json` | Full per-task details, generated code, pass/fail status, errors, timing, and resources. |
| `results/summary/summary_EXP_ID.json` | Summary metadata and the result list used for analysis. |
| `results/summary/summary_EXP_ID.csv` | Flat one-row CSV summary for spreadsheet/statistical tools. |
| `results/summary/comparison_ready_EXP_ID.json` | Compact cross-experiment comparison format. |

The runner saves these files before attempting any repository automation. If the
automatic branch/commit/push step is unavailable, it prints a warning but the
evaluation artifacts remain on disk.

## Regenerating Reports From Existing Results

After one or more `summary_*.json` files exist:

```bash
python evaluation/generate_report.py
```

This script reads `evaluation/results/summary/` and produces aggregate outputs
such as:

- `evaluation/results/tables/summary_table.csv`
- generated figures under `evaluation/results/figures/`
- `evaluation/results/report.md`

## Previously Generated Results

The current package includes results from earlier runs under:

- `results/raw/`
- `results/summary/`
- `results/tables/`

These files are part of the evaluation evidence. They should be interpreted as
historical experiment artifacts. To reproduce a table exactly, use the matching
experiment name, dataset, sample indices, attempts, translator/optimizer flags,
LLM backend, and environment recorded in the corresponding summary JSON.

## Reproducibility Checklist

For a reviewer or future researcher, record the following when rerunning:

1. Experiment name.
2. Dataset name and source.
3. Dataset limit or sample indices file.
4. Number of attempts.
5. Translator enabled/disabled.
6. Optimizer enabled/disabled.
7. Local or cloud LLM backend.
8. Model configuration if `--model_config` is used.
9. Hardware/runtime environment.
10. Generated files under `evaluation/results/`.

## Notes For Reviewers

- The package intentionally contains both code and previous result artifacts.
- Public datasets can be loaded online through Hugging Face, while local JSONL
  datasets can be supplied through `--dataset_local_path`.
- The raw result files may be large because they preserve per-task prompts,
  generated code, intermediate outputs, test results, timing, and resource
  measurements.
- Exact numerical reproduction can vary if the backend model, cloud service,
  dependency versions, or hardware differ from the original run.
